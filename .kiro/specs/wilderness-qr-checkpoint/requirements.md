# Requirements Document

## Introduction

A battery-powered ESP32-CAM checkpoint device deployed in wilderness environments with no internet connectivity. The device wakes on button press, reads a user's personal QR code, captures a proof-of-presence photo, and stores both the photo and a scan log entry to an SD card. Designed for outdoor events supporting up to 1000 users per device over an 8-hour window.

## Glossary

- **Device**: The checkpoint station built around the ESP32-CAM (AI-Thinker) board, including the OV2640 camera module, MicroSD card slot, DS3231 RTC module, KY-016 RGB LED, momentary push button, MiniUPS power management module, and lithium battery, all housed in a weatherproof enclosure. The ESP32-CAM board is the only compute component deployed in the field; the ESP32-CAM-MB programmer board is not included in the deployed Device.
- **ESP32-CAM**: The AI-Thinker ESP32-CAM development board (part of the ESP32-CAM-MB kit). Features an ESP32-S module, OV2640 2MP camera, MicroSD card slot, 4MB flash, WiFi/Bluetooth (WiFi unused in this project), and GPIO breakout pins. This is the board deployed in the field.
- **ESP32-CAM-MB**: The USB-C programmer/motherboard board (part of the ESP32-CAM-MB kit, CH340G USB-to-serial converter). Used during development only for flashing firmware and serial debugging. The ESP32-CAM plugs into this board via header pins. Not deployed in the field.
- **OV2640**: The 2-megapixel camera module included with the ESP32-CAM board. Supports JPEG capture at resolutions up to 1600x1200; this project uses VGA (640x480) for optimal file size and capture speed.
- **User**: A participant in an outdoor event who carries a personal QR code and interacts with the Device to register a checkpoint visit.
- **QR_Code**: A static QR code tied to a unique User_ID, generated by the Phone_App, presented by the User to the Device camera (OV2640) for scanning.
- **User_ID**: A unique alphanumeric identifier assigned to each User, encoded within the QR_Code. Format: 4-digit zero-padded number (e.g., "0001" through "1000").
- **Device_ID**: A unique identifier assigned to each Device. Format: 2-digit zero-padded number (e.g., "01" through "99").
- **Photo**: A JPEG image captured by the OV2640 camera at VGA resolution (~30KB), serving as proof-of-presence for a User at a checkpoint.
- **Scan_Log**: A CSV file stored on the SD_Card that records each scan event with User_ID, Device_ID, and RTC timestamp.
- **LED_Controller**: A software module in the Device firmware that manages the KY-016 RGB LED state machine (green, red, white, white/green alternating, blue, blue/green alternating, off). Implemented using the ESP32's built-in LEDC PWM peripheral — no additional controller hardware required. Uses 3 LEDC PWM channels (one per color) at 5kHz, 8-bit resolution. During SD card operations, only blue (GPIO 33) is available due to GPIO pin sharing. Not the onboard GPIO4 LED, which conflicts with the SD_Card interface.
- **KY-016**: A common cathode RGB LED module with built-in current-limiting resistors. Requires 3 PWM-capable GPIO pins on the ESP32-CAM and operates at 3.3V–5V. Colors are mixed via LEDC PWM duty cycle (0–255 per channel); white is achieved by driving all three channels at full duty.
- **MiniUPS**: A Type-C UPS power management module that handles lithium battery charging and regulated voltage output. Provides 3.3V and 5V outputs at up to 1.5A, supports simultaneous charge and discharge, and connects to a single-cell 3.7V lithium battery (18650 or LiPo). Charges via Type-C input. Used to power the ESP32-CAM and peripherals in the deployed Device.
- **Battery**: A single-cell 3.7V lithium battery (18650 or LiPo, 2000–3500mAh recommended) connected to the MiniUPS module. Provides power for the Device during field deployment. A 3000mAh cell provides comfortable margin for an 8-hour event with up to 1000 scans.
- **Sleep_Mode**: A low-power state the Device enters after a period of inactivity to conserve battery. Uses the ESP32's deep sleep mode with ext0 wake on button press.
- **Phone_App**: A mobile application that generates personal QR codes for Users, logs scans locally, and syncs to a backend when internet is available.
- **RTC**: The DS3231 real-time clock module connected to the ESP32-CAM via I2C. Provides accurate timestamps independent of network connectivity and maintains time via onboard coin cell battery backup.
- **SD_Card**: The MicroSD card inserted into the ESP32-CAM's built-in card slot, used for storing Photos and the Scan_Log. Formatted as FAT32. Note: the SD card interface shares GPIO pins with the camera on the ESP32-CAM, so the firmware must not access both simultaneously.

## Requirements

### Requirement 1: Device Wake from Sleep

**User Story:** As a User, I want to wake the Device by pressing a button, so that I can begin the checkpoint scan process.

#### Acceptance Criteria

1. WHILE the Device is in Sleep_Mode, THE Device SHALL wake and transition to the ready state within 2 seconds of a button press.
2. WHEN the Device transitions to the ready state, THE LED_Controller SHALL display solid green to indicate readiness to scan.
3. WHILE the Device is in Sleep_Mode, THE Device SHALL consume no more than 10mA of battery current.

### Requirement 2: QR Code Scanning

**User Story:** As a User, I want to show my personal QR code to the Device, so that the Device can identify me at this checkpoint.

#### Acceptance Criteria

1. WHILE the Device is in the ready state, THE Device SHALL continuously attempt to read a QR_Code from the camera feed.
2. WHEN the Device reads a QR_Code, THE Device SHALL extract the User_ID and validate that the User_ID matches the expected format (4-digit zero-padded alphanumeric string).
3. WHEN the Device reads a QR_Code with an invalid User_ID format, THE LED_Controller SHALL display solid red for 2 seconds, then return to solid green.
4. WHEN the Device reads a QR_Code with a valid User_ID, THE LED_Controller SHALL display solid white for 1 second to indicate acceptance.

### Requirement 3: Photo Countdown and Capture

**User Story:** As a User, I want a visual countdown before my photo is taken, so that I can prepare and face the camera.

#### Acceptance Criteria

1. WHEN the Device has accepted a valid QR_Code, THE LED_Controller SHALL display an alternating white/green blink pattern that accelerates from slow (500ms interval) to fast (100ms interval) over a 3-second countdown period, ending on solid white.
2. WHEN the countdown completes, THE LED_Controller SHALL remain solid white while the Device captures the Photo.
3. WHEN the Device captures a Photo, THE Device SHALL save the Photo to the SD_Card as a JPEG file at VGA resolution.
4. THE Device SHALL name each Photo file using the pattern DEVICE{Device_ID}_USER{User_ID}.jpg (e.g., DEVICE01_USER0042.jpg).
5. WHEN a Photo file already exists for the same User_ID on the same Device, THE Device SHALL overwrite the existing file with the new Photo.
6. WHEN the Photo has been captured, THE LED_Controller SHALL display solid blue during the SD_Card save operation, for a minimum of 1.5 seconds (or longer if the save takes longer). Only blue (GPIO 33) is used during SD operations due to GPIO pin sharing constraints.

### Requirement 4: Scan Logging

**User Story:** As an event organizer, I want each scan event recorded with a timestamp, so that I can verify when each User visited each checkpoint.

#### Acceptance Criteria

1. WHEN the Device captures a Photo for a User, THE Device SHALL append a record to the Scan_Log on the SD_Card containing the User_ID, Device_ID, and RTC timestamp.
2. THE Device SHALL format each Scan_Log record as a single CSV row with columns: User_ID, Device_ID, timestamp (ISO 8601 format).
3. WHEN the Device writes a Scan_Log record for a User who has been scanned before on the same Device, THE Device SHALL append a new record rather than overwrite the previous record.
4. IF the SD_Card is full or write fails, THEN THE Device SHALL display solid red on the LED_Controller for 5 seconds and return to the ready state without capturing a Photo.

### Requirement 5: Success Confirmation

**User Story:** As a User, I want clear visual confirmation that my checkpoint visit was recorded, so that I know I can move on.

#### Acceptance Criteria

1. WHEN the Device has saved a Photo and written a Scan_Log record, THE LED_Controller SHALL display an alternating blue/green pattern (100ms each) for 3 seconds.
2. WHEN the 3-second blue/green confirmation completes, THE LED_Controller SHALL return to solid green to indicate readiness for the next scan.

### Requirement 6: Inactivity Sleep

**User Story:** As an event organizer, I want the Device to conserve battery when not in use, so that the Device lasts the full 8-hour event window.

#### Acceptance Criteria

1. WHILE the Device is in the ready state, THE Device SHALL reset a 60-second inactivity timer each time a QR_Code scan completes (success or failure).
2. WHEN the inactivity timer reaches 60 seconds with no scan activity, THE Device SHALL enter Sleep_Mode.
3. WHEN the Device enters Sleep_Mode, THE LED_Controller SHALL turn off.

### Requirement 7: Storage Capacity

**User Story:** As an event organizer, I want the Device to handle up to 1000 users per event, so that all participants can register at each checkpoint.

#### Acceptance Criteria

1. THE Device SHALL support storing at least 1000 unique Photo files on the SD_Card (approximately 30MB at VGA resolution).
2. THE Device SHALL support a Scan_Log containing at least 2000 records (accounting for rescans).
3. WHEN the Device boots, THE Device SHALL verify that the SD_Card is present and writable, and display solid red on the LED_Controller if the SD_Card is missing or unwritable.

### Requirement 8: RTC Timestamp Accuracy

**User Story:** As an event organizer, I want accurate timestamps on scan records, so that I can reliably determine when each checkpoint visit occurred.

#### Acceptance Criteria

1. THE Device SHALL read the current time from the RTC module for each Scan_Log record.
2. IF the RTC module is unresponsive or returns an invalid time, THEN THE Device SHALL log the Scan_Log record with a fallback timestamp of "0000-00-00T00:00:00" and continue operation.

### Requirement 9: Photo File Naming and Format

**User Story:** As an event organizer, I want a consistent photo naming convention, so that I can easily match photos to users during post-event processing.

#### Acceptance Criteria

1. THE Device SHALL save each Photo as a JPEG file with the naming pattern DEVICE{Device_ID}_USER{User_ID}.jpg where Device_ID is a 2-digit zero-padded number and User_ID is a 4-digit zero-padded number.
2. THE Device SHALL capture Photos at VGA resolution (640x480 pixels).
3. THE Device SHALL produce Photo files no larger than 50KB each.

### Requirement 10: Scan Throughput

**User Story:** As an event organizer, I want the Device to handle a steady flow of participants, so that queues at checkpoints remain short.

#### Acceptance Criteria

1. THE Device SHALL complete a full scan cycle (QR read, countdown, photo capture, save, confirmation) in no more than 10 seconds from QR acceptance to return to ready state.
2. THE Device SHALL sustain an average throughput of 2 scans per minute over a continuous 30-minute period without degradation.

### Requirement 11: Phone App QR Code Generation

**User Story:** As a User, I want to generate my personal QR code on my phone, so that I can present it at any checkpoint Device.

#### Acceptance Criteria

1. THE Phone_App SHALL generate a QR_Code encoding the User_ID for each registered User.
2. THE Phone_App SHALL display the QR_Code at sufficient size and contrast for the Device camera to read at a distance of 10-30cm.
3. THE Phone_App SHALL store the generated QR_Code locally so it is available without internet connectivity.

### Requirement 12: Phone App Scan Logging and Sync

**User Story:** As a User, I want my phone to keep a local record of my scans and sync them when I have internet, so that I have a personal log of my checkpoint visits.

#### Acceptance Criteria

1. THE Phone_App SHALL log each scan event locally on the User's device with the timestamp and Device_ID (if communicated via QR response or manual entry).
2. WHEN internet connectivity is available, THE Phone_App SHALL sync local scan records to the backend.
3. IF sync fails, THEN THE Phone_App SHALL retain the local records and retry on next connectivity.

### Requirement 13: SD Card Data Retrieval

**User Story:** As an event organizer, I want to retrieve scan data from the Device after the event, so that I can process results and match photos to users.

#### Acceptance Criteria

1. THE Device SHALL store all Photos and the Scan_Log in a root-level directory on the SD_Card named after the Device_ID (e.g., /DEVICE01/).
2. THE Scan_Log SHALL be stored as a valid CSV file with a header row: "user_id,device_id,timestamp".
3. THE Device SHALL not require any special software to read the SD_Card contents — standard file system access (FAT32) SHALL be sufficient.
